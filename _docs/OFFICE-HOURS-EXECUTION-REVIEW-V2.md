## Problem Statement

Geodocs is correctly positioned as a definitive GEO/AEO knowledge hub for humans and AI agents.

The execution challenge is now simple and hard:
- Prove real utility with a small canonical set fast.
- Keep quality high through strict metadata and publish gates.
- Avoid process bottlenecks that kill weekly shipping cadence.

## Demand Evidence

User-provided evidence in this session:
- GEO/AEO information is fragmented across blogs and paid funnels.
- Users currently do: Google search -> AI search -> read -> compare -> still low trust.
- Estimated research cost is 15-20 hours/week for practitioners.

This is pre-product demand evidence, not revenue evidence.

## Status Quo

Execution state at review time:
- Strategy and design adaptation exist in geodocs.dev-ui.
- External content repo geodocs.dev-docs has been reset and scaffolded.
- Metadata schema, validator, and manifest generator exist.
- Sync from content repo to UI repo is operational in dry-run and real mode.
- Content count is intentionally zero after reset.

## Target User and Narrowest Wedge

Primary users for v1:
- SEO/GEO specialists
- Developers implementing technical GEO/AEO standards

Secondary users in v1 support path:
- Marketing managers/founders
- Content creators
- AI agents/bots as machine readers

Narrowest wedge:
- Fastest path from GEO/AEO question to trusted, actionable answer with machine-citable markdown.

## Constraints

- Keep Docusaurus architecture in geodocs.dev-ui.
- No full visual retheme in v1.
- Publish only from validated content contract.
- Deploy only after sync + analytics verification.

Explicit v1 exclusions:
- No 1000-page program execution in this sprint.
- No dedicated secondary-ICP portal.
- No major token-level visual redesign.

## Premises

PREMISES:
1. The right problem is execution reliability, not strategy rewrite.
2. A strict frontmatter contract is required to serve both humans and AI agents.
3. Weekly small-batch publishing is superior to monthly big-batch publishing for trust.
4. A single reviewer role without backup will become the top operational bottleneck.

Metadata contract (required per page):
- `title`
- `description`
- `primary_l1`
- `primary_l2`
- `artifact_type`
- `intent_stage`
- `owner`
- `review_date`

Definitions:
- Canonical page: production-ready page that passed metadata and technical review.
- `artifact_type`: page format classifier (checklist, template, script, matrix, sop, brief, reference).
- `intent_stage`: reader job stage (learn, plan, implement, measure, operate).
- Tactical artifact: actionable element on page (checklist, template, comparison table, script, or SOP).

## Approaches Considered

### Approach A: Foundation-First Sprint (Minimal Viable)
Summary: Spend 2 days on gates and ownership, then ship content in 5-day cadence.
Effort: M
Risk: Low-Med
Pros:
- High confidence in publish quality.
- Lower rework from metadata failures.
- Builds repeatable weekly operating system.
Cons:
- Slower visible output in first 48 hours.
- Requires strict role accountability.
Reuses:
- Existing schema, validator, manifest, sync script, UI docs structure.

### Approach B: Content-First Parallel Push
Summary: Publish immediately while fixing gates in parallel.
Effort: M
Risk: High
Pros:
- Fast short-term visible content output.
- Earlier real-reader feedback.
Cons:
- Higher risk of metadata drift and rollback events.
- Team confusion on ownership boundaries.
Reuses:
- Same assets, but with process debt.

### Approach C: Platform-First Buildout (Ideal Long-Term, Not v1)
Summary: Expand automation and editorial tooling before shipping significant content.
Effort: L
Risk: Med
Pros:
- Better long-term scale posture for 1000 pages.
- Strong compliance and review automation.
Cons:
- Overbuild risk before proving wedge demand.
- Delays actual utility for users.
Reuses:
- Current foundation plus new tooling layers.

## Recommended Approach

Choose Approach A.

Why:
- It keeps execution risk controlled while still shipping within one week.
- It directly targets the known failure mode: inconsistent quality under speed pressure.
- It protects brand trust, which is core for a definitive reference hub.
