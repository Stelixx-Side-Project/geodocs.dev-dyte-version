## Problem Statement

The strategic direction is clear: geodocs should become the definitive GEO/AEO knowledge hub for humans and AI agents.

The current question is execution readiness:
- Are strategy, IA, schema, and pipeline aligned enough to start content production now?
- What is still missing before a reliable publish loop can run weekly?

## Demand Evidence

Current evidence is directional but meaningful:
- Existing GEO traffic and repeat reading behavior indicate pull.
- User reports quality vacuum in current GEO/AEO resources.
- Practical need is immediate: teams need references they can execute from, not trend commentary.

## Status Quo

Current state across repos:
- geodocs.dev-ui contains strategy, design system adaptation, and content pipeline doc.
- geodocs.dev-docs has been reset and now contains taxonomy skeleton, schema, validation script, and manifest builder.
- Content inventory is intentionally empty (`articles.json` count = 0), which is acceptable for from-scratch reset.

## Target User and Narrowest Wedge

Primary v1 users:
- SEO/GEO specialists
- Developers implementing GEO/AEO technical standards

Secondary v1 support:
- Marketing managers/founders
- Content creators
- AI agents as machine readers

Narrowest wedge:
- Fastest path from GEO/AEO question to trusted, actionable answer with machine-citable structure.

## Constraints

- Keep existing Docusaurus architecture.
- Do not perform full visual retheme in v1.
- Ship from-scratch content system with strict metadata contract.
- Keep publish loop safe: validate first, sync second, deploy last.

Explicit v1 exclusions:
- No 1000-page expansion program execution in this sprint.
- No advanced theme-token redesign.
- No secondary-ICP dedicated portal in v1 sprint.

## Premises

1. Strategy quality is not the blocker anymore, execution discipline is.
2. A strict metadata contract is required for both human navigation and AI-agent readability.
3. Weekly shipping cadence beats large-batch publishing for trust and quality.

Metadata contract (required per page):
- `title`
- `description`
- `primary_l1`
- `primary_l2`
- `artifact_type`
- `intent_stage`
- `owner`
- `review_date`

## Approaches Considered

### Approach A: Harden Pipeline First (Minimal Viable)
Summary: Freeze content expansion for 2-3 days, close operational gaps, then publish.
Effort: M
Risk: Low
Pros:
- Minimizes bad publishes.
- Establishes repeatable quality gate before scale.
- Preserves current strategy without rewrite.
Cons:
- Slower visible content output in the next 48 hours.
- Requires process ownership discipline.
Reuses:
- Existing schema, validator, manifest builder, and sync script.

### Approach B: Parallel Build (Current Momentum)
Summary: Publish content while tightening pipeline in parallel.
Effort: M
Risk: Med-High
Pros:
- Faster short-term output.
- Immediate testable content surface.
Cons:
- High risk of inconsistent metadata and quality drift.
- Potential for rework if gates change mid-stream.
Reuses:
- Existing docs + scripts, but with temporary process debt.

### Approach C: Platform-First Expansion
Summary: Extend tooling, add advanced automation, then start publishing.
Effort: L
Risk: Med
Pros:
- Strong long-term automation posture.
- Better scalability controls for 1000-page target.
Cons:
- Overbuild risk before proof of editorial loop.
- Delays actual knowledge-hub utility.
Reuses:
- Existing foundation, but introduces larger upfront engineering.

Note:
- Approach C is explicitly out of v1 sprint scope.

## Recommended Approach

Choose Approach A.

Why:
- The strategy is ready, execution reliability is the true bottleneck.
- A short hardening sprint unlocks predictable weekly publishing.
- This path protects trust, which is the core asset of a reference hub.

## Success Criteria

Execution criteria for this sprint window (2026-04-22 to 2026-04-29):
- Pipeline gate criteria:
  - Validation passes in geodocs.dev-docs on every PR.
  - Manifest regenerates and syncs without manual patching.
  - Sync only imports allowed folders.
- Product criteria:
  - Minimum 8 canonical pages published (4 fundamentals, 4 execution).
  - Every page includes required frontmatter and one tactical artifact.
- Measurement criteria:
  - 3 baseline analytics events firing in staging and production:
    - `homepage_cta_click`
    - `article_artifact_interaction`
    - `role_path_progression`

Execution owner map:
- Content owner: writes and updates canonical pages.
- Metadata reviewer: checks frontmatter contract and taxonomy consistency.
- Technical reviewer: validates sync and build safety.
- Deployer: merges sync PR and runs deploy flow.
- Analytics validator: confirms event firing before production release.

## Distribution Plan

Content distribution channel:
- Author and review in geodocs.dev-docs.
- Sync into geodocs.dev-ui via scripted import.
- Publish via existing Docusaurus + Cloudflare deployment flow.

CI/CD expectation:
- Content repo validates metadata and manifest on PR/push.
- UI repo runs sync dry-run and site build checks before deploy.

Allowed-folder sync policy (explicit):
- Import only `docs/fundamentals`, `docs/strategy`, `docs/execution`, `docs/proof`, `docs/operations`, and `docs/_manifests/articles.json` from source repo.
- Reject all other folders from automated import.
